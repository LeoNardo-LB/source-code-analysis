## Context

OpenCode（https://github.com/sst/opencode）是一个 139k Star 的开源 AI 编码代理，采用 TypeScript + Bun 运行时，Turborepo monorepo 管理 18 个子包。核心引擎位于 `packages/opencode/src/`，包含 40+ 功能模块（Agent、Provider、Tool、Session、LSP、MCP、TUI 等）。

目标读者是有 Java 后端经验、学过 JS/TS 的开发者。最终文档将推送到 GitHub 仓库，通过手机浏览阅读。

当前项目目录结构为空，需要同时拉取源码和创建文档体系。

## Goals / Non-Goals

**Goals:**

- 创建一套完整的、移动端友好的 OpenCode 源码学习指南
- 以"消息请求生命周期"为主线，让读者理解端到端的完整流程
- 每篇文档自包含，不看源码也能理解模块运转机制
- 大量使用 Mermaid 图表（架构图、时序图、流程图）作为主要表达手段
- 用伪代码 + 详细注释说明核心逻辑，真实源码作为可选深入锚点
- 源码目录（`opencode/`）与文档目录（`docs/`）严格分离
- 覆盖四大学习目标：架构理解、Agent 模式、TS 工程实践、二次开发

**Non-Goals:**

- 不做 OpenCode 的修改或贡献（纯学习研究）
- 不做代码翻译（伪代码说明逻辑，不逐行翻译源码）
- 不覆盖测试、CI/CD 等非核心模块的深入分析
- 不做视频教程或交互式内容
- 文档不追求实时同步上游更新，基于拉取时的最新 tag 版本

## Decisions

### D1: 文档主线选择 — 按请求生命周期（方案 B）

**选择**: 以"用户输入一条消息 → Agent 推理 → Tool 执行 → 结果输出"的完整链路为主线组织文档。

**替代方案**:
- 按架构分层（自上而下）：结构清晰但故事线弱，读者难以理解模块间协作
- 按学习曲线渐进：入门友好但同一概念可能在多篇重复

**理由**: 用户明确表示要"面对整个流程来分析"，方案 B 的故事线最强，且天然展示模块间协作关系。融合方案 C 的渐进深度思想，每篇内部按"概念 → 图解 → 伪代码 → 深入"四层递进。

### D2: 图表技术选型 — Mermaid

**选择**: 全部使用 Mermaid 图表，依托 GitHub 原生渲染能力。

**替代方案**:
- ASCII Art：兼容性最强但表达力弱，复杂图表难以维护
- 外部图片（PNG/SVG）：需要额外生成工具，增加维护成本
- PlantUML：GitHub 不原生支持

**理由**: GitHub 原生支持 Mermaid 渲染，手机端直接可看，维护成本低，表达力足够（支持流程图、时序图、架构图、状态图、甘特图等）。

### D3: 目录分离策略

**选择**: `opencode/` 存放源码，`docs/` 存放文档，两者同级。

**替代方案**:
- 文档放在源码仓库内的 docs 目录：耦合源码，更新时冲突风险高
- 独立文档仓库：需要额外维护仓库关系

**理由**: 分离后文档可独立版本管理，源码作为只读参考，互不干扰。读者可以只关注 docs/ 目录。

### D4: 文档模板设计 — 四层递进结构

**选择**: 每篇文档统一采用"架构位置图 → 问题定义 → 运转流程图 → 核心伪代码 → 设计决策 → 源码锚点"的模板。

**理由**: 前三层（图 + 伪代码）确保不看源码也能理解，后两层（设计决策 + 源码锚点）满足深入探索需求。轻量 Java 类比穿插在"设计决策"中。

### D5: 源码版本策略 — 最新 tag

**选择**: 拉取 opencode 仓库的最新 tag 版本，而非 dev 分支。

**理由**: tag 版本更稳定，文档内容不会因上游频繁变动而过时。后续需要时可以基于新 tag 更新。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 上游代码更新导致文档过时 | 文档基于固定 tag 版本，开头注明对应版本号，后续可增量更新 |
| Mermaid 复杂图表在手机端渲染不佳 | 控制单张图的节点数量（≤15），复杂系统拆分为多张子图 |
| 伪代码与真实实现产生偏差 | 每个伪代码块标注对应源码路径，读者可交叉验证 |
| 18 篇文档工作量巨大 | 采用"先骨架后血肉"策略，先出路线图 + 架构总览 + 入门篇，后续逐步补充 |
| Effect 框架对 Java 开发者理解门槛高 | 专题文档（13 篇）+ 附录 A（TS 速查）降低入门门槛 |
